# Pirsig: AST Quality Gates

**Subsystem:** `crates/pirsig`
**Role:** Deterministic code evaluation and hallucination prevention.

## Mechanism

Pirsig acts as the final gatekeeper before any code generated by an LLM is permitted to touch the filesystem. It operates on the fundamental principle that **probabilistic generation must be constrained by deterministic validation.**

When the Codernic agent proposes a patch or a new file:
1. **AST Parsing:** Pirsig intercepts the payload and parses the code into an Abstract Syntax Tree (AST) using `tree-sitter`.
2. **Deterministic Rules Engine:** It evaluates the AST against a set of hardcoded and project-specific rules:
    - **Import Validation:** Checks every imported module/symbol against the `Ragtime` index. If the LLM hallucinated an import that doesn't exist, it flags a violation.
    - **Export Discipline:** Ensures the file adheres to the project's export rules (e.g., preventing deep relative imports or enforcing barrel files).
    - **Syntax Integrity:** Ensures the code is syntactically valid and compiles/transpiles.
3. **Severity Scoring:** Violations are categorized:
    - `info` / `warn`: The code passes but the agent is notified.
    - `red` / `black`: The code is strictly blocked. A "black" violation (like a hallucinated import) causes a hard rejection. The file is not written to disk, and the LLM receives the error trace to try again.

## Why it matters (Our Philosophy)

The industry trend is to treat LLMs like human developers: let them write code, and put a human in the loop to review the Pull Request. This doesn't scale for multi-agent autonomous workflows. If an agent writes 400 lines of code but hallucinates a library, the human wastes time untangling the mess.

By catching structural errors at the AST level *before* they reach disk, Pirsig turns a probabilistic output into a deterministic guarantee. This drastically reduces the "review burden" on the developer, allowing them to trust the autonomous system.
